# Go 轻量主机面板 MVP 设计规格

**日期**：2026-04-17  
**状态**：已评审（brainstorming 三节确认）  
**范围**：裸机 Linux、Nginx + 模板 + Go 内嵌 ACME（HTTP-01）、站点与反代、HTTPS（Let’s Encrypt 类 CA）

---

## 1. 背景与仓库关系

- 本规格描述 **独立 Go 项目**（类宝塔轻量子集），与当前仓库 `weaviate-web-client`（Vue 前端）**无代码耦合**；若落地实现，建议使用 **独立仓库** 或 Monorepo 中独立目录，由本文件追溯需求来源即可。

---

## 2. 目标与非目标

### 2.1 目标（MVP）

- 在 **单台裸机 Linux** 上，通过 **Web/API** 管理「站点」：**域名**、**反代上游**、**HTTPS 开关**。
- 底层由 **Nginx** 提供入口；证书由 **Go 内嵌 ACME**（如 CertMagic）申请与续期，首期仅 **HTTP-01**。
- 配置变更经 **`nginx -t` 校验** 后再 **`reload`**；失败则 **不切换** 已生效配置。

### 2.2 非目标（本期不做）

- PHP、MySQL、FTP、计划任务、多租户复杂 RBAC、集群与多节点同步。
- TLS-ALPN-01、DNS-01（可列为后续迭代）。

---

## 3. 技术路线（已定：丙）

- **Nginx**：入口与反代；面板 **仅管理** 约定目录下的片段（如 `panel.d`），不接管整台机器全部 Nginx 配置。
- **证书**：**Go 内嵌 ACME**（推荐 CertMagic 或等价库），与面板同一进程或同一代码库，状态集中。
- **备选已否决**：纯 certbot（甲）、纯 Caddy 替代 Nginx（乙）——不作为 MVP 主路径。

---

## 4. 架构与模块

| 模块 | 职责 |
|------|------|
| API / Web | 登录与会话、站点 CRUD、触发应用配置、错误展示 |
| Config 引擎 | 期望状态（DB）→ 生成 Nginx 片段（建议 **每站点一文件**） |
| ACME 子系统 | HTTP-01 申请/续期；挑战文件落盘；与 Nginx `alias` 配合 |
| Nginx 驱动 | 原子写临时文件 → `nginx -t` → 成功则替换并 `reload` |
| 状态存储 | SQLite（默认）或 PostgreSQL；站点、证书元数据、审计日志 |
| 安全边界 | 非 root 常驻、特权通过 sudo/setcap/白名单、面板默认仅本机监听 |

---

## 5. 数据流与目录约定

### 5.1 保存站点主路径

1. 提交站点（域名、上游、是否 HTTPS 等）→ **输入校验**（格式、冲突）。
2. **持久化** DB，状态可为 `pending` / `applied` / `failed`（实现时固定枚举名）。
3. **渲染** Nginx 片段至 **临时路径**，并维护 **HTTP-01** 所需 `/.well-known/acme-challenge/`（见 6 节）。
4. **`nginx -t`**：**失败**则不替换正式文件、不 reload，DB 记 `failed` 并保存 `stderr` 摘要；**成功**则 **rename** 覆盖正式配置并 **`nginx -s reload`**。
5. 证书就绪且配置合法后，站点记为 `applied`。

### 5.2 默认目录（可通过配置文件覆盖）

| 用途 | 默认路径 | 说明 |
|------|-----------|------|
| 面板数据（DB 等） | `/var/lib/panel/` | 运行用户可写 |
| 生成中的 Nginx 片段 | `/etc/nginx/panel.d/` | 主配置 `include` 此目录；与人工其它 `conf` 分离 |
| 证书与私钥 | `/var/lib/panel/certs/<domain>/` | 目录 700，私钥 600，属主为面板用户 |
| 审计与日志 | DB 表 + 或 `/var/lib/panel/logs/` | 含操作者与 `nginx -t` 摘要 |

### 5.3 失败与回滚

- **`nginx -t` 失败**：不 reload；不覆盖已生效文件；返回错误与日志摘要。
- **`reload` 失败**（少见）：保留上一版配置；记录告警。
- **证书申请失败**：见 **7. 证书策略**。

---

## 6. ACME HTTP-01 与 Nginx

- **首期仅 HTTP-01**。
- **推荐实现**：挑战响应为 **磁盘文件** + Nginx **`alias`** 到面板写入的目录（如 `/var/lib/panel/acme-challenge/`），与常见运维习惯一致、易抓包排查。
- **顺序**：先保证 **80 端口** 可从公网访问 challenge 路径，再向 CA 发起申请。

---

## 7. 证书策略（明确行为）

- **若用户开启 HTTPS 但证书尚未就绪**：生成的 Nginx 配置 **仅包含 HTTP `server` 块**（反代业务照常），**不写** `ssl_certificate` / `listen 443 ssl` 等指令，直至证书申请成功后再生成 HTTPS 块并 reload。
- **若证书申请失败**：站点保持 **HTTP 可访问**（在配置合法前提下）；DB 标记证书失败原因；管理员可在 UI 中重试申请。

---

## 8. 安全与权限

- **运行用户**：专用系统用户（如 `panel`），**不以 root 常驻**。
- **特权**：`nginx -t`、`reload`、写 `/etc/nginx/panel.d/` 等通过 **sudoers 白名单** 或 **setcap** + 目录权限组合实现；规格实现阶段在 `README` 中给出 **最小权限** 示例命令（两行级 sudo 或等价方案）。
- **面板监听**：默认 **`127.0.0.1:<端口>`**；公网访问需 **SSH 隧道** 或 **防火墙**；若必须对公网开放面板，须 **HTTPS + 强密码**，速率限制；2FA 为后续迭代。
- **认证**：首期 **单管理员**；密码使用 **bcrypt 或 argon2**；会话优先 **HttpOnly Cookie**，降低 XSS 窃取风险；Web 表单配合 **CSRF 防护** 或 **SameSite** 策略。

---

## 9. API 形态

- **REST + JSON**，前缀 **`/api/v1/`**。
- **资源示例**：`GET/POST/PATCH/DELETE /api/v1/sites`，`GET /api/v1/health`（进程、DB、可选只读配置检查）。
- **错误体**：`code`、`message`、可选 `detail`（含 `nginx` 校验摘要）。

---

## 10. Web UI（MVP）

- 最小功能：**站点列表**、**增删改**、**触发证书**、**展示最近一次错误**。
- 技术选型：**Go 模板 + htmx** 或 **独立 SPA** 二选一，**不阻塞** 后端 API 交付。

---

## 11. 测试策略

- **单元**：模板渲染（给定模型 → 期望片段）；域名校验。
- **集成**：临时目录 + **假 `nginx` 脚本** 模拟 `nginx -t` 成功/失败，断言「失败不 reload」。
- **CI**：不占真实 80；ACME 使用 **staging** 或 **mock**。

---

## 12. 验收标准（MVP）

1. 管理员可新增站点：域名解析到本机后，**HTTP 反代**可用。
2. 开启 HTTPS 后，在 **HTTP-01** 可达条件下可签发证书，**HTTPS** 可用。
3. 故意错误配置时 **`nginx -t` 失败**，**线上配置不被破坏**。
4. 面板未登录无法改站；关键操作有 **审计记录**。
